# data/eval/ — розмічений eval-датасет для калібрування детектора

Призначення: **калібрування порога й ваг детектора** (Фаза 2). Детектор
перемикає розкладку, лише коли альтернативна інтерпретація суттєво ймовірніша.
Щоб налаштувати поріг (`threshold(len)`) і ваги LM/словника, потрібен розмічений
набір із обома класами.

## Найважливіше: НЕГАТИВНИЙ клас — головний

**Precision > recall.** Хибне перемикання (зіпсувало легітимний текст) дратує
користувача сильніше, ніж пропуск (не перемкнуло крякозябри). Тому негативний
клас (`should_switch=false`) — навмисно багатший і різноманітніший за позитив, і
саме на ньому міряємо хибні спрацювання. Метрика прийняття калібрування — низький
false-positive rate на негативі при притомному recall на позитиві.

## Файл

`dataset.jsonl` — один приклад на рядок (JSONL). Генерується з `generate.py`
(тримаємо генератор у git заради відтворюваності, як і для LM/словників).
Перегенерувати: `python data/eval/generate.py`.

## Схема (стабільна; кожен рядок — рівно ці 5 полів)

```json
{"text": "ghbdsn", "typed_layout": "en", "intended_layout": "uk", "should_switch": true, "category": "pos_en_for_uk"}
```

- `text` — рядок, який **фактично на екрані** (для позитиву — крякозябри; для
  негативу — легітимний текст).
- `typed_layout` — розкладка, активна на момент набору (`"en"` / `"uk"`).
- `intended_layout` — розкладка, яку користувач **насправді мав на увазі**.
- `should_switch` — **головна мітка**. `true` → треба перемкнути й перенабрати;
  `false` → НЕ чіпати.
- `category` — група для аналізу метрик по зрізах (див. нижче).

Інваріанти: для позитиву `typed_layout != intended_layout` і
`should_switch=true`; для негативу `typed_layout == intended_layout` і
`should_switch=false`.

## Категорії

**POSITIVE (`should_switch=true`)** — реальні слова в неправильній розкладці, в
обидва боки (фізична позиція клавіш спільна, тому `привіт`↔`ghbdsn`):
- `pos_en_for_uk` — хотів uk, активна була en → на екрані латинські крякозябри.
- `pos_uk_for_en` — хотів en, активна була uk → на екрані кирилиця (напр.
  `hello`→`руддщ`). Слова різної довжини, включно з короткими (детектору
  найважче).

**NEGATIVE (`should_switch=false`)** — легітимний текст, який перемикати не можна:
- `uk_legit` / `en_legit` — коректні слова/фрази у своїй розкладці.
- `short_ambiguous` — короткі реальні слова обох мов (`по`, `не`, `to`, `is`):
  мало контексту → LM невпевнений → найризикованіші для хибного перемикання.
- `code` — токени коду (`git`, `npm`, `fn`, `impl`, `const`): латиниця,
  низькочастотна у природній мові.
- `brand_nick` — бренди, нікнейми, юзернейми.
- `url_path_email` — URL, email, шляхи, IP, `@mention`, `#hashtag`.
- `mixed` — змішані uk/en фрази (`зроби git push`).
- `acronym` — абревіатури (`США`, `API`).
- `slang` — сленг/вигуки (`лол`, `imho`).
- `alphanumeric` — буквено-цифрові токени (`win10`, `2fa`).

## ПРИВАТНІСТЬ (залізно)

**Тільки синтетичні дані. Нуль реальних секретів** — паролів, токенів, особистих
даних, реальних адрес. URL/email/шляхи — лише плейсхолдери (`example.com`,
`/usr/...`, `user@example.com`). Те саме правило, що для фікстур — див.
`fixtures/CLAUDE.md`. TypoFix бачить увесь ввід, тож реальний запис у git злив би
секрети назавжди.

## Розмітка

`should_switch` розмічається людиною за задумом (як треба), **не** виводиться
поточним детектором — інакше калібрування циклічне (підлаштується саме під себе).
Це той самий принцип незалежної розмітки, що й для `*.expected.jsonl`
(`docs/TESTING_STRATEGY.md` §2).
