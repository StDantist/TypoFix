# TypoFix — Стратегія тестування й автономної перевірки

> Мета: я (AI-розробник) маю змогу **сам**, як реальний користувач, відтворити
> кейс, знайти баг і виправити його. Це визначає архітектуру (ARCHITECTURE.md
> §1, §3, §4-virtual).

## 1. Чесні межі автономії

Автономія **повна для core-логіки** і **напівавтономна для платформного шару** —
важливо це не плутати:

- **Повністю автономно (без людини, кожен коміт):** уся логіка рішень — у
  `typofix-core` як **чиста детермінована функція** `step(state, event, ctx) ->
  actions` (без ОС, годинника, random). Через `typofix-platform-virtual`
  тестується повний конвеєр E2E *на рівні логіки* — як побачив би користувач.
- **Напівавтономно (разовий грант дозволу людиною):** реальний платформний шар.
  Virtual platform тестує **нашу модель ОС, а не саму ОС**, тож платформні баги
  (тайм-аут хука, message-pump, гонка перемикання розкладки, UIPI, auto-disable
  tap на macOS, розбіжність `ToUnicodeEx` з TOML, autocomplete-поля) вона **не
  ловить**. Для них — рівень 4 (real-OS харнес). На macOS потрібен одноразовий
  ручний грант Accessibility; далі прогони автоматичні.

**Не обманюймо себе:** більшість найкаверзніших багів живе саме в платформному
шарі. Тому real-OS харнес будуємо рано (Фаза 0.5 спайк + Фаза 5), а не наприкінці.

## 2. Рівні тестів (піраміда)

### Рівень 0 — Unit (core, миттєві, тисячі)
- `layout_mapper`: scancode↔символ для кожної розкладки, шифт, апостроф, `ґ`.
- `lm`: score відомих uk/en слів > score крякозябрів; межові випадки.
- `dict`: наявність/відсутність, відмінки.
- `detector`: правильна мова + confidence на наборі пар (`привіт`↔`ghbdsn`).
- `rules`/`exceptions`: вето, регекси, самонавчання.
- `replacer`: коректний план (N backspace, послідовність, регістр, Caps).
- `undo`: відкат + commit у винятки.

### Рівень 1 — Integration (core + virtual platform, headless E2E)
Через `typofix-platform-virtual`: «друкуємо» послідовність у віртуальне вікно,
проганяємо повний конвеєр, читаємо віртуальний текстовий буфер і перевіряємо
результат — **точно як побачив би користувач**, але без реальної ОС.

Приклади сценаріїв:
- Набір `ghbdsn ` → буфер містить `привіт ` (перемкнуло + перенабрало).
- Набір у виключеному застосунку → нічого не змінено.
- WASD-послідовність у грі → ігнор.
- Скасування → текст відкочено + слово у винятках; повтор не перемикає.
- Слово з цифрами/URL/нікнейм → не чіпає.

### Рівень 2 — Golden / replay фікстури
- Сесії натисків у `fixtures/*.jsonl` — **лише синтетичні або скрабовані**.
  TypoFix бачить увесь ввід, тож реальні записи могли б містити паролі/секрети →
  у репозиторій вони **не потрапляють** (privacy-first, ARCHITECTURE §8).
- Очікуваний вихід `*.expected.jsonl` **розмічається незалежно** (людиною або
  окремим інструментом), а **не генерується поточним движком** — інакше тест
  циклічний: ловитиме лише зміни, а не відхилення від задуму.
- Регресії ловляться автоматично; кожен новий баг → нова фікстура («баг як тест»).

### Рівень 3 — Property-based (fuzz)
- `proptest`: для будь-якого слова `w` у мові L: набравши його в *неправильній*
  розкладці, движок має відновити саме `w` (інваріант round-trip).
- Перенабір ніколи не залишає буфер у неконсистентному стані.
- Движок ніколи не входить у нескінченний цикл перемикань.

### Рівень 4 — Реальна ОС (автоматизований real-OS харнес)
Єдине, що ловить платформні баги. Будуємо **рано** (спайк у Фазі 0.5, повноцінно
у Фазі 5), не наприкінці.
- **Windows:** харнес-вікно або Notepad; драйвимо реальний ввід і зчитуємо текст
  через **UI Automation**. Перевіряємо реальний ланцюг: хук (з message-pump) →
  Unicode-ін'єкція → перемикання розкладки → перенабір. Тести в CI на
  Windows-runner'і автоматично.
- **macOS:** аналог через Accessibility API; разовий ручний грант дозволу, далі
  автоматично.
- Покриває саме те, що virtual не може: тайм-аути, гонки, UIPI, autocomplete,
  `ToUnicodeEx`-розбіжності, інпут-лаги.

## 3. Як я перевіряю як «реальний користувач» (цикл автономії)

1. Пишу/міняю код.
2. `cargo test` — рівні 0–3 (детерміновано, секунди).
3. Для UX-кейсу пишу integration-сценарій (рівень 1) мовою користувача:
   «набрав X у застосунку Y → очікую Z».
4. Якщо баг — додаю фікстуру (рівень 2), що його відтворює, потім фікшу до
   зеленого.
5. Періодично — рівень 4 (real-OS харнес) для платформеної впевненості.
6. UI перевіряю через `tauri-driver`/WebDriver (клік по налаштуваннях, стани).

## 4. Інструменти

- Тест-раннер: вбудований `cargo test` + `cargo nextest` (швидше).
- Property: `proptest`.
- Покриття: `cargo llvm-cov` (ціль: core ≥ 90%).
- Lint/format: `clippy -D warnings`, `rustfmt` — обов'язкові в CI.
- UI E2E: `tauri-driver`/WebDriver (Tauri = WebView2/WKWebView, не звичайний браузер).
- CI: GitHub Actions, матриця Windows + macOS; рівень 4 (real-OS) — окремий job
  на відповідному runner'і.

## 5. Definition of Done для будь-якої фічі

- [ ] Unit-тести на нову логіку (рівень 0).
- [ ] Хоча б один integration-сценарій користувача (рівень 1).
- [ ] Якщо фікситься баг — додано regression-фікстуру (рівень 2).
- [ ] `clippy` чистий, `fmt` застосовано, покриття не впало.
- [ ] Якщо чіпали платформений шар — пройшов smoke на реальній ОС (рівень 4).

## 6. Тестова інфраструктура для збірки рано

`typofix-platform-virtual` і `typofix-replay` — **частина MVP інфраструктури**,
не «потім». Вони будуються у фазі 0 (див. DEVELOPMENT_PLAN), бо без них немає
автономії.
