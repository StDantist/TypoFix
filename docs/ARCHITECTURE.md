# TypoFix — Архітектура

> Передумова: прочитано CONCEPT.md. Цей документ описує *як* збудовано систему.

## 1. Принцип поділу

```
┌──────────────────────────────────────────────────────────────┐
│ UI (ui/)  — Tauri webview: трей + вікно налаштувань            │
│            завантажується лише при відкритті → ~0 RAM у спокої │
└───────────────────────────┬──────────────────────────────────┘
                            │ Tauri IPC (commands/events)
┌───────────────────────────▼──────────────────────────────────┐
│ src-tauri/  — оркестрація: життєвий цикл, трей, конфіг,        │
│               під'єднання core ↔ platform, потоки              │
└───────────────────────────┬──────────────────────────────────┘
                ┌───────────┴───────────┐
                ▼                       ▼
┌───────────────────────────┐ ┌────────────────────────────────┐
│ typofix-core (чистий Rust) │ │ typofix-platform (trait + impl) │
│ БЕЗ залежностей від ОС     │ │ Windows / macOS / Virtual       │
│ детермінований, тестований │ │ хук, ін'єкція, розкладка, вікно │
└───────────────────────────┘ └────────────────────────────────┘
                ▲
┌───────────────┴───────────┐
│ typofix-data               │
│ розкладки, LM, словники    │
└────────────────────────────┘
```

**Золоте правило:** уся логіка прийняття рішень — у `typofix-core` як **чисті
функції без сайд-ефектів і без ОС-залежностей**. Платформа лише постачає події й
виконує дії. Це робить **core-логіку** повністю автономно тестованою; платформний
шар тестується напівавтономно (real-OS харнес із разовим грантом дозволу) —
чесні межі автономії описані в TESTING_STRATEGY.md.

## 3. `typofix-core` — двигун рішень

### 3.1 Модель як state machine
Ядро — детермінована машина: на вхід потік `KeyEvent` + `Context`, на вихід
послідовність `Action`. Жодного I/O, жодного часу з годинника (час передається у
події), жодної випадковості.

```
fn step(state: &mut EngineState, ev: KeyEvent, ctx: &Context) -> Vec<Action>
```

- `InputEvent::{ Key(KeyEvent), MouseClick, FocusChange(WindowInfo), CaretMove }`
  — у core йдуть **усі** події, не лише клавіші, бо буфер інвалідується не тільки
  натисками (див. §3.4).
- `KeyEvent { scancode, vk, down/up, modifiers, timestamp, is_synthetic, is_autorepeat }`
- `Context { active_app, layout_id, settings }`
- `Action::{ None, SwitchLayout(id), DeleteChars(n), TypeUnicode(string),
  CommitException(word), … }`
  - **Важливо:** перенабір тексту йде через `TypeUnicode(string)` (готові
    символи), а **не** через повтор scancode у новій розкладці. `SwitchLayout`
    — окрема дія, потрібна лише щоб *подальший* набір користувача був правильним.
    Розділення усуває гонку з асинхронним перемиканням розкладки (див. §4).

### 3.2 Підмодулі core
- **`buffer`** — буфер натисків поточного «слова» **per-window** (scancodes +
  регістр + модифікатори). Скидається на межі слова **та інвалідується** на
  будь-якій нелінійній події (див. §3.4) — це критично, інакше `DeleteChars(n)`
  зітре чужий текст.
- **`layout_mapper`** — `scancode + layout → символ` і навпаки; будує
  альтернативні інтерпретації послідовності в інших розкладках.
- **`lm`** — символьна n-gram модель: `score(word, lang) -> logprob`. Компактна,
  завантажується з `typofix-data`.
- **`dict`** — швидка перевірка наявності слова (FST/трай) для бусту впевненості.
- **`detector`** — поєднує lm+dict+mapper: для слова рахує найкращу мову/розкладку
  і `confidence`; вирішує, чи перемикати, за порогом.
- **`rules`** — білий список, регекси, контекстні правила; вето на перемикання.
- **`exceptions`** — динамічні винятки (самонавчання зі скасувань).
- **`replacer`** — будує план заміни `DeleteChars(n) + SwitchLayout(id) +
  TypeUnicode(text)` зі збереженням регістру/Caps та апострофів. Текст іде як
  Unicode (не повтор scancode) — щоб не залежати від моменту перемикання розкладки.
- **`undo`** — стек останніх перемикань для відкату + commit у винятки.

### 3.3 Алгоритм рішення (на межі слова)
1. Узяти буфер слова (scancodes).
2. Для кожної ввімкненої мовної пари побудувати інтерпретацію в її розкладці.
3. `score = w1*lm_logprob + w2*dict_hit − penalties(rules)`.
4. Якщо найкраща інтерпретація ≠ поточна і `Δscore > threshold(len)` і немає вето
   правил → `DeleteChars + SwitchLayout + TypeUnicode`.
5. Інакше — нічого.

**`threshold(len)` — поріг залежить від довжини слова.** Короткі слова (2–3
літери) валідні в обох мовах одночасно (`по`/`gj`, `не`/`yt`), тож для них поріг
високий (або взагалі не чіпати без сильної переваги); для довших — нижчий. Єдиний
глобальний поріг дав би або багато хибних спрацювань на коротких, або пропуски.
Ваги `w1,w2`, penalties і крива `threshold(len)` **калібруються на розміченому
датасеті з негативним класом** (див. §3.5 і DEVELOPMENT_PLAN Фаза 2), а не задаються
«на око».

### 3.4 Інвалідація буфера (коректність — блокер)
Буфер слова відображає текст безпосередньо *перед курсором*. Будь-що, що розриває
цей зв'язок, **мусить скинути буфер**, інакше перенабір зітре не те:
- Клік мишею (зміна позиції курсору).
- Навігаційні клавіші: стрілки, Home/End, PageUp/PageDown.
- Будь-яке виділення тексту.
- Зміна фокуса вікна/контролу (`FocusChange`) — буфер ще й per-window.
- Ручний Backspace/Delete коригує буфер (не повна інвалідація, а синхронізація).
- Auto-repeat (утримання клавіші) — дедуплікація, щоб не флудити буфер.
- Сполучення з Ctrl/Alt/Win (Ctrl+C, Ctrl+S) — **не** потрапляють у буфер як літери.

### 3.5 Контроль хибних спрацювань
Пріоритет №2 (влучність) тримається не на алгоритмі сам по собі, а на калібруванні:
- **Негативний клас** у датасеті — легітимний текст, який перемикати НЕ можна:
  сленг, нікнейми, код, технічні терміни, змішані uk/en речення.
- Метрики: **precision** (не перемикати зайве) важливіша за recall — хибне
  перемикання дратує більше, ніж пропуск.
- `threshold(len)` + penalties правил калібруються під цільові precision/recall.

## 4. `typofix-platform` — абстракція ОС

```rust
trait Platform {
    fn keyboard_hook(&self) -> EventStream;        // постачає KeyEvent
    fn inject(&self, keys: &[InjectKey]);          // SendInput / CGEventPost
    fn set_layout(&self, id: LayoutId);            // ActivateKbLayout / TISSelect
    fn active_window(&self) -> WindowInfo;         // процес, exe-шлях, fullscreen
    fn current_layout(&self) -> LayoutId;
}
```

| Можливість | Windows | macOS |
|---|---|---|
| Хук | `SetWindowsHookEx(WH_KEYBOARD_LL)` | `CGEventTap` (+ Accessibility) |
| Ін'єкція | `SendInput` | `CGEventPost` |
| Розкладка | `ActivateKeyboardLayout` / `WM_INPUTLANGCHANGEREQUEST` | `TISSelectInputSource` |
| Активне вікно | `GetForegroundWindow`→PID→exe | `NSWorkspace.frontmostApplication` |
| Перенабір тексту | `SendInput` з `KEYEVENTF_UNICODE` | `CGEventKeyboardSetUnicodeString` |
| Розкладка для мапінгу | `ToUnicodeEx(scancode, HKL)` | `UCKeyTranslate` через TIS |
| Дозволи | (антивірус може лаяти хук) | Accessibility обов'язково |

**Підводні камені платформи, які треба врахувати в реалізації (не в дизайні core):**

*Windows:*
- `WH_KEYBOARD_LL` callback має **тайм-аут** (`LowLevelHooksTimeout`); перевищиш —
  ОС **тихо знімає хук**. Звідси правило «hook thread робить мінімум» (§5).
- Потік хука **зобов'язаний мати message-pump** (`GetMessage`), інакше callback
  не викликається взагалі.
- **UIPI:** неелевований процес не отримує подій і не може інжектити у вікна
  вищої цілісності (Task Manager, regedit, admin-cmd). У таких вікнах TypoFix
  просто не працює — це задокументоване обмеження (див. §10), не баг.
- Мапінг розкладки брати через `ToUnicodeEx` з конкретним `HKL` (з прапором, що
  не змінює стан dead-key буфера), а не лише з вшитого TOML — TOML лише fallback/тест.

*macOS:*
- Система **вимикає tap** (`kCGEventTapDisabledByTimeout`/`ByUserInput`); треба
  ловити цю подію й **повторно вмикати** tap.
- **Secure Input mode** (поля паролів тощо): tap не отримує натисків — це
  системне обмеження, не баг (§10).
- Accessibility-грант прив'язаний до підпису бінаря → під час розробки перебілд
  може скидати дозвіл (тертя для real-OS тестів — див. TESTING_STRATEGY §2, рівень 4).

`typofix-platform-virtual` реалізує цей самий trait повністю в пам'яті
(віртуальна клавіатура + віртуальний текстовий буфер + віртуальне «активне
вікно») — для автономних E2E-тестів. **Застереження:** virtual тестує *нашу
модель* ОС, а не саму ОС; платформні баги вище він не ловить — для них є рівень 4.

## 5. Потокова модель (продуктивність)

- **Hook thread:** callback робить мінімум — кладе `KeyEvent` у lock-free канал.
  Жодних обчислень тут (інакше лагає вся ОС).
- **Engine thread:** читає канал, ганяє `core::step`, надсилає `Action` у
  platform injector.
- **UI/Tauri thread:** трей + IPC, прокидається лише на взаємодію.
- Синтетичні події (наш власний перенабір) позначаються `is_synthetic`, щоб хук
  їх ігнорував (немає циклу).

## 6. Формати даних (мовна-агностичність)

- **Розкладки:** основне джерело мапінгу `scancode → символ` у рантаймі — **сама
  ОС** (`ToUnicodeEx`/`UCKeyTranslate`) для реально активної розкладки. TOML
  (`data/layouts/{id}.toml`, мапа `scancode → {normal, shift, altgr}`) — fallback,
  еталон для тестів і опис розкладок, яких немає в системі. AltGr і dead-keys
  обробляються через ОС-шлях, бо статична `normal/shift` мапа їх не покриває.
- **LM:** `data/lm/{lang}.bin` — серіалізована n-gram модель (компактна).
- **Словники:** `data/dicts/{lang}.fst` — FST для O(довжина) перевірки.
- **Конфіг користувача:** `settings.json` у per-OS app-data; містить мовні пари,
  поріг, виключення, правила, гарячі клавіші, винятки самонавчання.
- Дані вбудовуються в бінар через `typofix-data` (embed) з можливістю override
  з диска.

## 7. Конфігурація і сховище

- Windows: `%APPDATA%\TypoFix\`; macOS: `~/Library/Application Support/TypoFix/`.
- Atomic write (tmp + rename). Налаштування пристрою окремо від «навчених» даних.
- Версіонування схеми конфігу + міграції.

## 7а. Виключення застосунків і папок

- Виключення за **іменем процесу** і **повним шляхом до exe** (`GetForegroundWindow`
  →PID→exe / `frontmostApplication.bundleURL`).
- «Виключення папки» = **префікс шляху до exe**: будь-який процес, запущений
  з-під обраної теки (рекурсивно), вимикає TypoFix. Це покриває кейс «папка з
  іграми». *Не* означає «робочу директорію документа» — її для довільного GUI
  достовірно не визначити, і це поза цілями.
- Авто-пауза у fullscreen/exclusive-input (детект на платформі).

## 8. Безпека й приватність (закласти рано)

- **Буфер натисків — лише в пам'яті, ніколи не персиститься.** TypoFix за
  природою бачить увесь ввід; жодного логування натисків на диск у проді.
- **Password/secure-поля:** на macOS Secure Input фізично глушить tap (ОК); на
  Windows немає універсального API детекту — застосовуємо евристику (клас вікна
  `EDIT` з `ES_PASSWORD`, відомі апи) і за можливості **не буферимо** такі поля.
- **Фікстури:** записані сесії натисків (`fixtures/*.jsonl`) — **лише синтетичні
  або скрабовані**; реальні секрети/паролі в репозиторій не потрапляють (див.
  TESTING_STRATEGY §2).
- Code signing (Windows) + notarization (macOS) — відкладено (TODO релізу), бо
  EDR/Gatekeeper можуть лаяти хук/ін'єкцію. Для особистого користування не потрібно.
- Жодних мережевих дозволів у рантаймі (privacy-first).

## 9. Відкриті технічні питання

- Точна стратегія перенабору в полях з агресивним autocomplete/IME-композицією
  (чати/IDE), де видимий текст ≠ кількість натисків → `DeleteChars(n)` ненадійний.
  Потребує спайку на реальній ОС (DEVELOPMENT_PLAN Фаза 0.5).
- Надійний детект password-полів на Windows.

## 10. Обмеження й платформні non-goals

- **IME-мови (китайська, японська, корейська) — НЕ підтримуються.** Модель
  `scancode → символ` принципово не описує IME-композицію. «Фундамент для інших
  мов» стосується лише **алфавітних розкладкових** мов (більшість кириличних і
  латинських). CJK — явний non-goal.
- **Елевовані вікна на Windows (UIPI)** — TypoFix не працює у вікнах вищої
  цілісності, поки сам не елевований (елевацію не робимо — псує «легку безпечну апку»).
- **Поля паролів / Secure Input** — навмисно не обробляються.
- **3+ активні розкладки** одночасно: ціль перемикання обирається як найкраща
  мовна пара за score; якщо неоднозначно — не перемикати (краще пропуск, ніж
  хибне перемикання).
