# TypoFix — План розробки

> Передумова: прочитано CONCEPT.md, ARCHITECTURE.md, TESTING_STRATEGY.md.
> Це повний поетапний план до **повноцінного продукту** (не MVP). Фази задають
> порядок реалізації; кожна фаза має чіткий Definition of Done. Чеклісти —
> робочий трекер; познач `[x]` по ходу.

## Принципи виконання

- Кожна фаза закінчується **зеленими тестами** відповідних рівнів (див.
  TESTING_STRATEGY.md) і чистим `clippy`.
- Платформена абстракція + віртуальна платформа будуються **рано** (фаза 0–1),
  щоб уся подальша робота була автономно тестованою.
- Windows — перша ціль; macOS-порт — окрема фаза, але trait готовий з початку.
- «Баг → спершу regression-фікстура, потім фікс».

## Фаза 0 — Каркас і інфраструктура автономії

**Мета:** робочий скелет + можливість тестувати все headless.

- [ ] Cargo workspace зі структурою крейтів (ARCHITECTURE §2).
- [ ] Tauri-застосунок: трей-іконка, порожнє вікно налаштувань, autostart-заглушка.
- [ ] `typofix-platform`: визначити trait `Platform` і спільні типи

      з `TypeUnicode`, `WindowInfo`, `LayoutId`).
- [ ] `typofix-platform-virtual`: повна реалізація trait у пам'яті (віртуальні
      клавіатура/текст/вікно/розкладка).
- [ ] `typofix-replay`: формат фікстур (`*.jsonl`), record/replay API.
- [ ] CI: GitHub Actions (Win+macOS), `cargo test`, `clippy -D warnings`, `fmt`.
- [ ] Скелет `typofix-core::step` (поки повертає `Action::None`) з першим
      integration-тестом через virtual platform.

**DoD:** `cargo test` зелений; CI працює; можна «надрукувати» у віртуальне вікно
і прочитати буфер.

## Фаза 0.5 — Платформний спайк на реальній Windows (де-ризокування)

**Мета:** перевірити НАЙРИЗИКОВІШІ платформні припущення ДО побудови решти, бо
вони можуть змінити дизайн. Це короткий брудний прототип, не продакшн-код.

- [ ] Глобальний `WH_KEYBOARD_LL` з message-pump: переконатися, що події йдуть і
      хук не знімається тайм-аутом.
- [ ] `SendInput` з `KEYEVENTF_UNICODE`: інжекція готового рядка `привіт` у Notepad.
- [ ] `ActivateKeyboardLayout`/`WM_INPUTLANGCHANGEREQUEST` + одразу `TypeUnicode`:
      підтвердити, що Unicode-шлях НЕ страждає від гонки перемикання розкладки.
- [ ] `DeleteChars(n)` у реальному полі з autocomplete (адресний рядок браузера,
      пошук) — виміряти, де `n` натисків ≠ видимі символи; зафіксувати стратегію.
- [ ] `ToUnicodeEx(scancode, HKL)` для uk/en — звірити з вшитим TOML.
- [ ] Перевірити поведінку у вікні Task Manager (UIPI) — підтвердити обмеження §10.

**DoD:** письмовий висновок у `docs/SPIKE_FINDINGS.md`; за потреби скориговано
модель `Action`/`Context` у ARCHITECTURE.md ПЕРЕД фазами 2–3.

## Фаза 1 — Дані: розкладки, мовні моделі, словники

**Мета:** усе мовно-залежне як дані + завантажувачі.

- [ ] `data/layouts/`: uk (ЙЦУКЕН з апострофом/`ґ`) та en (QWERTY) у TOML
      (scancode → normal/shift/altgr) — як еталон/fallback; рантайм-мапінг бере
      з ОС (`ToUnicodeEx`).
- [ ] `typofix-data`: парсер розкладок, embed у бінар + override з диска.
- [ ] Корпуси uk/en (відкриті джерела: Wikipedia dump, brown-uk тощо) → скрипт
      тренування n-gram LM → `data/lm/{lang}.bin`. Українська: усі словоформи
      (флективна мова), не лише леми.
- [ ] **Розмічений датасет для калібрування** (`data/eval/`): позитивний клас
      (крякозябри↔правильні) + **негативний клас** (легітимні uk/en/змішані/сленг/
      нікнейми/код, які перемикати НЕ можна). Скрипт генерації синтетики + ручне
      розмічення; **без реальних секретів**.
- [ ] Словники uk/en у FST (`data/dicts/{lang}.fst`).
- [ ] `typofix-core::layout_mapper`, `lm`, `dict` + unit-тести (рівень 0).

**DoD:** `lm.score("привіт","uk") > lm.score("ghbdsn","uk")`; mapper round-trip
коректний для всіх клавіш; покриття цих модулів ≥ 90%.

## Фаза 2 — Двигун розпізнавання й рішення

**Мета:** правильно визначати неправильну розкладку.

- [ ] `buffer`: накопичення слова на scancodes, регістр; скидання на межі слова
      **+ інвалідація** на MouseClick/навігаційних клавішах/FocusChange/виділенні/
      auto-repeat (ARCHITECTURE §3.4). Модифіковані сполучення не йдуть у буфер.
- [ ] `detector`: альтернативні інтерпретації + scoring (lm+dict) + confidence.
- [ ] **`threshold(len)`** — поріг, залежний від довжини слова (короткі — суворіше).
- [ ] Калібрування ваг/порога на датасеті з негативним класом → цільові
      precision/recall.
- [ ] Вибір найкращої мовної пари; за неоднозначності (3+ розкладок) — не перемикати.
- [ ] Unit + integration сценарії: позитивні ТА негативні приклади.
- [ ] Property-тест round-trip (рівень 3) — з урахуванням не-бієкції uk↔en.

**DoD:** на повному eval-датасеті (позитив+негатив, не лише 200 пар) досягнуто
цільових **precision ≥ ціль / recall ≥ ціль** (precision пріоритетніша);
**нуль перемикань** на негативному класі понад поріг.

## Фаза 3 — Перенабір (replacement) і скасування

**Мета:** реально виправляти вже набране.

- [ ] `replacer`: план `DeleteChars(n) + SwitchLayout(id) + TypeUnicode(text)`,
      збереження регістру/Caps/апострофа. Текст — Unicode, не повтор scancode.
- [ ] `undo`: стек перемикань, відкат, commit слова у винятки (самонавчання).
- [ ] Інтеграція з virtual platform: повний E2E `ghbdsn ` → `привіт `.
- [ ] E2E інвалідації: клік/стрілки/фокус у середині слова → перенабір НЕ чіпає
      чужий текст (захист від втрати даних).
- [ ] Захист від циклів (синтетичні події ігноруються) + інтерлівінг із швидким
      набором користувача під час перенабору.

**DoD:** E2E перенабору, скасування й інвалідації зелені; fuzz не ламає
консистентність і не видаляє чужий текст.

## Фаза 4 — Правила, винятки, контекст застосунків

**Мета:** мінімум хибних спрацювань + керованість.

- [ ] `rules`: білий список, регекси (цифри/`@`/`/`/URL/email/шляхи), вето.
- [ ] `exceptions`: персистентні винятки + інтеграція зі скасуванням.
- [ ] Контекст активного вікна в `Context` (процес/exe-шлях/fullscreen).
- [ ] Виключення за процесом, exe-шляхом і **папкою (префікс exe-шляху)** +
      авто-пауза у fullscreen.
- [ ] Детект password/secure-полів → не буферити (евристика на Windows; Secure
      Input на macOS глушить tap автоматично).

**DoD:** виключені застосунки/папки не чіпаються; правила дають вето; самонавчання
працює end-to-end.

## Фаза 5 — Платформа Windows (реальна)

**Мета:** працює на справжній Windows.

- [ ] `typofix-platform-windows`: `WH_KEYBOARD_LL` (з message-pump),
      `SendInput`+`KEYEVENTF_UNICODE`, `ActivateKeyboardLayout`/
      `WM_INPUTLANGCHANGEREQUEST`, `ToUnicodeEx`, `GetForegroundWindow`→PID→exe,
      детект fullscreen, mouse/focus події → `InputEvent`.
- [ ] Потокова модель: hook thread (мінімум) → канал → engine thread → injector.
- [ ] Обмеження UIPI задокументоване й коректно деградує (елевовані вікна — no-op).
- [ ] **Real-OS харнес** (Win UI Automation: керуємо вводом, читаємо текст
      Notepad/харнес-вікна) — автоматизований, у CI на Windows-runner'і.

**DoD:** реальний набір крякозябрів у Notepad виправляється; real-OS харнес
зелений у CI; RAM у спокої в бюджеті, CPU на hot-path ≈ 0; немає інпут-лагів.

## Метрики успіху (числові бюджети)

- **Влучність:** precision ≥ 0.99 (майже нуль хибних перемикань), recall ≥ 0.90
  на eval-датасеті uk/en — фінальні цілі уточнюються після baseline у Фазі 2.
- **Хибні спрацювання** на негативному класі: ~0.
- **Hook hot-path:** обробка події у callback < 1 мс (далеко від тайм-ауту ОС).
- **Латентність перенабору:** < 50 мс від межі слова до завершення заміни.
- **RAM у спокої:** < 30 МБ (без відкритого вікна налаштувань).
- **CPU у спокої:** ≈ 0% (подієва модель, без полінгу).
